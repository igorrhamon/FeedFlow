## Incidente e correção (2026-07-09/10)

Durante a implementação paralela das 6 workstreams acima, subagentes rodando
em worktrees isolados (por design) acabaram, por engano, commitando e — em
um caso — pushando diretamente para o `main` do GitHub, sem PR nem review.
Cronologia:

1. Detectado quando 4 das 6 workstreams apareceram como commits na `main`
   compartilhada (`7bb5499`, `1f4ed09`, `f15471a`, `3446fd1`) em vez de nos
   worktrees isolados que receberam.
2. Confirmado via `git ls-remote` que o `main` real no GitHub já estava 3
   commits à frente do esperado (`f15471a`).
3. Nada foi perdido — todo o conteúdo foi preservado numa branch de resgate
   local antes de qualquer correção.
4. A pedido do usuário, o `main` do GitHub foi corrigido com um commit de
   revert (`2fed2d4`) — não uma reescrita de histórico — trazendo-o de volta
   ao estado do PR #11.
5. As 4 workstreams reveladas pelo incidente (WS-3, WS-2, WS-6, WS-17) foram
   reaplicadas via cherry-pick (não merge de branch — um merge direto da
   branch de resgate descartaria silenciosamente o conteúdo já revertido,
   por causa de como o git resolve "um lado mudou, o outro não" em relação
   ao ancestral comum).
6. Durante a reaplicação, apareceram bugs reais (API errada do drift,
   configuração incorreta da tabela virtual FTS5 que quebrava todo
   insert/update em `WorkItems`, teste com timing flaky) — corrigidos antes
   de abrir o PR #12.

Causa raiz: instruções passadas aos subagentes não garantiram isolamento
efetivo — pelo menos 4 dos 6 agentes operaram no checkout compartilhado em
algum momento (provavelmente por `cd` explícito para o caminho absoluto do
repo, em vez de permanecer no worktree atribuído). Recomendação para
próximas rodadas: verificar explicitamente, antes de qualquer commit de
subagente, que `git rev-parse --show-toplevel` aponta para o worktree
esperado, não para o checkout principal.
